## Overview

What follows is a walk-through of my current terminal-centered research environment, reproduced here in enough detail that it could serve as documentation. I write academic prose across English, Spanish, Nahuatl, and Portuguese; my deliverables are book chapters, journal articles, edited primary sources, and lecture notes. All of it is composed in Markdown, compiled to PDF through Pandoc with a Chicago note-bibliography CSL, and managed through a thin layer of shell, Vim, and Sway configuration. The aesthetic is amber-on-black or Nord-on-near-black, monospace throughout, no images and no mouse — and I'll defend that aesthetic as a substantive choice rather than a cosmetic one in the manifesto at the end.

The pieces covered below:

- **Sway** — the Wayland window manager that glues the rest together through floating terminal windows
- **The Lurking deck** — a Python-curses launcher that opens on every shell
- **`rf`** — a `find`+`fzf` recent-files browser with text-only previews, including ANSI-colored ASCII for images
- **DAVID'S IRE** — my Vim configuration, including Startify with a Dávila/`cowsay` randomizer, vim-pandoc, fzf integration, a BibTeX-key-to-PDF function, and conceal-based list rendering
- **Ranger** — file manager with `jp2a` previews
- **Newsboat** — RSS reader with `rdrview`+`lynx` rendering
- **Email** — `aerc`, `matcha`, and `cmdg`
- **Zathura** — PDF viewer with a Nord palette

A note on how the configuration came together. A non-trivial fraction of what follows — Vim functions, shell pipelines, Ranger `scope.sh` exit codes, the `lurking.py` launcher itself — was assembled iteratively in conversation with large language models. I do not consider myself a programmer. I have not memorized vimscript syntax, and I have a working but partial grasp of bash. The fact that I can specify what I want in plain English, get a working snippet, test it, refine it, and end up with a personal environment shaped to my work has materially changed what is worth attempting. It used to be that customizing a tool at this depth required either a long apprenticeship or a willingness to copy other people's dotfiles wholesale. That barrier has effectively collapsed. The configuration below is mine in the sense that the choices and the use cases are mine, and I have read and edited every line — but I would not have written most of it from scratch.

## 1. Sway: floating terminals as connective tissue

Everything else in this setup runs inside Alacritty, and most of it runs inside a *floating* Alacritty window managed by Sway. The pattern is the same in every case: a global keybinding spawns a fixed-size, centered terminal that runs one specific TUI program.

```sway
## DHC's PERSONAL KEYBINDINGS FOR I3/SWAY
# Define the variable for floating terminal windows
set $term_float alacritty --class floating_terminal
for_window [app_id="floating_terminal"] floating enable, resize set 1300 700, move center

bindsym $mod+alt+r exec $term_float -e newsboat
bindsym $mod+alt+f exec firefox
bindsym $mod+alt+v exec alacritty -e vim
bindsym $mod+alt+o exec $term_float -e ranger
bindsym $mod+alt+n exec /home/dhcolmenares/Scripts/nv.sh
bindsym $mod+alt+l exec alacritty -e lurking

# Running cmdg requires bash -l -c so that EDITOR/PAGER environment
# variables are picked up.
bindsym $mod+alt+m exec alacritty -e bash -l -c "/usr/local/bin/cmdg -config ~/.cmdg/cmdg-uci.conf"
bindsym $mod+alt+p exec foot -e bash -l -c "/usr/local/bin/cmdg"
```

Two details worth flagging.

The `--class floating_terminal` flag tags the window so Sway's `for_window` rule matches it specifically. Regular Alacritty windows tile as usual; only those launched through `$term_float` float at the configured size and position. The `floating enable` predicate in the rule is mandatory — without it, the resize/move directives apply but the window still tiles.

The `bash -l -c "…"` wrapper appears on every command that depends on environment variables, aliases, or shell functions. The `-l` flag spawns a *login* shell, which sources `~/.profile`, `~/.bash_profile`, and the rest of the login chain. Without it, custom paths and shell functions defined in those files are not visible to the spawned process. This matters because several of the commands below — most importantly the `rf` shell function — exist only as bash function definitions, not as standalone scripts on `$PATH`.

## 2. The Lurking Deck

![The Lurking deck on shell launch](images/lurking.png)
*The Lurking deck. Amber-on-black, UCI header, dated, with the day's destinations.*

The Lurking deck is a Python `curses` application named after the opening screen of Infocom's 1987 text adventure *The Lurking Horror*. It runs on every interactive shell launch via a guarded line in `~/.zshrc`:

```bash
[[ $- == *i* ]] && python3 ~/bin/lurking.py
```

The `[[ $- == *i* ]]` test ensures it only fires for interactive shells, so cron jobs, scripts, and SSH non-interactive sessions are not affected.

The deck reads its menu from a plain text file at `~/.config/lurking/items`. The format is one item per line, four pipe-delimited fields:

```
KEY | LABEL | DESCRIPTION | COMMAND
```

`KEY` is a single-character hotkey (or `--` for none), `LABEL` is the short name shown in the left column, `DESCRIPTION` is the longer string shown to the right, and `COMMAND` is any shell command. Lines beginning with `#` and blank lines are ignored.

The current items file:

```
# ~/.config/lurking/items
# ──────────────────────────────────────────────────────────────────
1 | ELECTRONIC MAIL    | Fetch personal messages                  | alacritty -e bash -l -c "matcha"
1 | UCI MAIL           | Fetch messages from the aether           | alacritty -e bash -l -c "/usr/local/bin/cmdg -config ~/.cmdg/cmdg-uci.conf"
2 | VIM                | Open Classics paper                      | alacritty -e bash -l -c "vim"
3 | RECENT FILES       | Browse recent files                      | alacritty -e bash -l -c "rf1"
R | RECENT FILES (BG)  | Browse recent files with fzf, detached   | setsid alacritty -e bash -l -c "rf1"
4 | READ NEWS          | Read RSS feed                            | alacritty -e bash -l -c "newsboat"
5 | NOTEPAD            | Scratch pad                              | /home/dhcolmenares/Scripts/nv.sh
4 | RANGER             | File manager                             | alacritty -e bash -l -c "ranger"
```

A few things to note. The `setsid` wrapper on the `R` entry detaches the spawned terminal from the deck's process group, so the launcher can exit cleanly without leaving an orphaned child. Repeated keys (`1`, `4`) are intentional: the deck cycles through entries with the same hotkey on repeated presses.

![Close-up of items.txt](images/lurking-items.png)
*The items file, edited in Vim. New entries are picked up at the next deck launch.*

The deck does not need to be restarted when the items file changes; it is re-read at every launch. Because the launcher is keybound to `Mod+Alt+L` in Sway, I can refresh it instantly from anywhere.

For commands like `vim`, `ranger`, `newsboat`, and `matcha` — TUIs that take over the terminal — the wrapping pattern is `alacritty -e bash -l -c "…"`. This spawns a fresh terminal with a login shell that runs the program. The deck's own terminal is left undisturbed.

## 3. `rf`: Recent Files With ASCII Previews

The single most-used entry on the deck is `RECENT FILES`. Behind it is a shell function defined in `~/.bashrc`:

```bash
rf() {
  find ~/Documents ~/Downloads ~/Desktop ~/research -mtime -3 -type f \
    \( -name "*.pdf" -o -name "*.md" -o -name "*.jpg" -o -name "*.jpeg" \
       -o -name "*.txt" -o -name "*.docx" \) 2>/dev/null \
  | fzf --prompt="recent > " \
        --preview='case {} in
          *.pdf)
            pdftotext -l 1 {} - 2>/dev/null | batcat --color=always --style=plain --language=txt ;;
          *.md)
            batcat --color=always --style=plain --language=md {} ;;
          *.jpg|*.jpeg)
            jp2a --width=80 --colors {} ;;
          *.docx)
            pandoc -t plain {} 2>/dev/null | batcat --color=always --style=plain --language=txt ;;
          *)
            batcat --color=always --style=plain {} 2>/dev/null ;;
        esac' \
        --preview-window=right:55% --height=90% --ansi \
  | xargs -r xdg-open
}
```

The pipeline has three stages.

**Find.** The `find` command scans four working directories — `~/Documents`, `~/Downloads`, `~/Desktop`, `~/research` — for files matching one of six extensions and modified within the last three days (`-mtime -3`). The `2>/dev/null` suppresses permission errors on subdirectories I cannot read. The three-day window is empirically the right size: it consistently captures whatever I actually need to come back to without flooding the picker.

**Filter.** The output pipes into `fzf` with a per-extension preview command. Each branch of the `case` statement uses a different rendering strategy:

- **PDF.** `pdftotext -l 1` extracts only the first page; `batcat --color=always --style=plain --language=txt` adds light syntax coloring. This is enough to read a paper's title, abstract, and author block in the preview pane.
- **Markdown.** `batcat` with `--language=md` applies Markdown-aware highlighting: headings, lists, inline code, and emphasis become visible.
- **JPEG.** `jp2a --width=80 --colors` converts the image to ANSI-colored ASCII art at 80 columns wide. The `--ansi` flag on `fzf` allows the escape codes through to the preview pane. This sidesteps the entire question of image protocols (kitty, sixel, Überzug) — I am on Wayland under Sway, where the X11-only options do not apply, and the maintained Wayland fork `ueberzugpp` introduces dependencies I prefer to avoid. Rendering images as text means the same preview machinery works everywhere.
- **DOCX.** `pandoc -t plain` produces a clean text rendering of the document. No need to launch LibreOffice to remember what is in a peer review.
- **Default.** Plain `batcat` with auto-detected language for everything else.

**Open.** The selected path is piped through `xargs -r xdg-open`, which dispatches by MIME type: PDFs to Zathura, Markdown to Vim, images to my configured image viewer.

![ASCII preview of a JPEG inside fzf](images/rf-jp2a.png)
*A JPEG previewed as ANSI-colored ASCII inside fzf. Wayland-compatible, terminal-only, no image protocol required.*

The deck launches `rf` (technically `rf1`, a wrapper that exits after a single selection) inside a fresh Alacritty process, so the picker appears in its own window. The whole loop — `Mod+Alt+L` to surface the deck, digit to launch `rf`, fuzzy substring to filter, Enter to open — takes about three seconds.

## 4. DAVID'S IRE: Vim as an Integrated Research Environment

![Vim opening on Startify](images/vim-startify.png)
*Startify with a randomized Gómez Dávila aphorism in a `cowsay` frame, MRU files below, and four pinned bookmarks.*

The center of gravity of the workflow is Vim. My `~/.vim/.vimrc` opens with an ASCII banner reading **DAVID'S IRE** — Integrated Research Environment, but also an honest acknowledgment of what Vim periodically produces in its user. What follows is the full account of how it is configured, reading roughly in the order the file does.

The functional plugins are `vim-pandoc` and `vim-pandoc-syntax` for the actual editing experience, `fzf`/`fzf.vim` for file and content search, and `goyo.vim` for distraction-free writing. The rest are colorschemes, rotated according to mood and ambient lighting.

### Startify, with a Dávila aphorism randomizer

Startify is the home page Vim displays when launched without an argument. Two things make it useful for me. First, four pinned bookmarks at the top:

```vim
let g:startify_bookmarks = [
  \ {'a': '~/Documents/nvdata/index-writ.md'},
  \ {'b': '~/Documents/nvdata/courses-sandbox.md'},
  \ {'c': '~/Documents/nvdata/LOG.md'},
  \ {'d': '~/.vim/.vimrc'}
\ ]
let g:startify_session_dir = '~/.vim/sessions'
let g:startify_files_number = 5
let g:startify_fortune_use_unicode = 1
map ,s :Startify <CR>
```

Pressing `a` opens the master index of writing projects, `b` the courses scratchpad, `c` the daily log, and `d` the `.vimrc` itself. `,s` brings Startify back at any time mid-session.

Second, the rotating header. I keep a few hundred aphorisms by Nicolás Gómez Dávila — the Colombian aphorist whose *escolios* are some of the most condensed prose in Spanish — in `~/.config/nvim/davila_aphorisms.txt`, one per line. A small shell pipeline picks a random line and frames it with `cowsay` to produce the Startify header on each launch:

```vim
" Pick a random Dávila aphorism and frame it cowsay-style for the
" Startify header. The text file is plain UTF-8, one aphorism per line.
let g:startify_custom_header =
  \ systemlist('shuf -n 1 ~/.config/nvim/davila_aphorisms.txt | cowsay -W 60')
```

`shuf -n 1` selects a uniformly random line; `cowsay -W 60` wraps the result at sixty columns inside the standard cow's ASCII speech bubble. `systemlist()` returns the multi-line output as a Vim list, which Startify accepts directly as a header. Each Vim launch produces a different aphorism in a different frame. *Sólo es noble lo que dura.*

### Pandoc preview chain

Two custom commands compile the current Markdown buffer to PDF and open or refresh Zathura:

```vim
command! MDPreview execute "silent !pandoc --citeproc --csl=/home/dhcolmenares/Documents/imprenta/chicago-note-bibliography.csl.xml % --pdf-engine=pdfroff -t pdf -o /tmp/vim-markdown-preview.pdf && (pkill -HUP zathura || zathura /tmp/vim-markdown-preview.pdf &)" | redraw!

autocmd FileType markdown nnoremap <F12> :MDPreview<CR>

command! ViewPDF execute "silent !pandoc --citeproc --csl=/home/dhcolmenares/Documents/imprenta/chicago-note-bibliography.csl.xml % -o /tmp/vim-markdown-preview.pdf && (pkill -HUP zathura || zathura /tmp/vim-markdown-preview.pdf &)" | redraw!

nnoremap <F11> :ViewPDF<CR>
```

The two differ in PDF engine: `MDPreview` uses `pdfroff` (faster, troff-based); `ViewPDF` uses Pandoc's default LaTeX engine (slower, more complete). Both apply the same Chicago note-bibliography CSL. `pkill -HUP zathura` sends a hangup signal to any running Zathura, which Zathura interprets as "reload current document"; the `||` fallback opens a fresh instance if none is running.

### BibTeX key → PDF

This is the function I would miss most if it were taken away:

```vim
function! OpenPDFFromBibtexKey()
    let l:bibtex_key = expand('<cword>')
    let l:clean_key = substitute(l:bibtex_key, '^@', '', '')
    let l:pdf_dir = expand('/home/dhcolmenares/Documents/DHC-BIBTEX-LIBRARY')
    let l:find_cmd = 'find ' . shellescape(l:pdf_dir) . ' -iname "*' . l:clean_key . '*.pdf" -type f'
    let l:matches = systemlist(l:find_cmd)

    if len(l:matches) == 0
        echo 'No PDF found matching: ' . l:clean_key
        return
    elseif len(l:matches) == 1
        let l:pdf_path = l:matches[0]
        call system('zathura ' . shellescape(l:pdf_path) . ' &')
        echo 'Opened: ' . fnamemodify(l:pdf_path, ':t')
    else
        echo 'Multiple matches found:'
        for i in range(len(l:matches))
            echo (i + 1) . ': ' . fnamemodify(l:matches[i], ':t')
        endfor
        let l:choice = input('Choose file (1-' . len(l:matches) . '): ')
        let l:idx = str2nr(l:choice) - 1
        if l:idx >= 0 && l:idx < len(l:matches)
            let l:pdf_path = l:matches[l:idx]
            call system('zathura ' . shellescape(l:pdf_path) . ' &')
            echo 'Opened: ' . fnamemodify(l:pdf_path, ':t')
        else
            echo 'Invalid selection'
        endif
    endif
endfunction

command! OpenPDF call OpenPDFFromBibtexKey()
nnoremap <leader>op :call OpenPDFFromBibtexKey()<CR>
```

Place the cursor on any BibTeX key in the text — `@vermeule2017`, for example — and press `,op`. The function strips the `@`, runs a `find -iname` over `~/Documents/DHC-BIBTEX-LIBRARY/` for any PDF whose filename contains the key, and either opens the result in Zathura or, if there are several matches, prompts for a choice. With a bibliography of a few thousand entries, this turns "let me check that page reference" from a context switch into two seconds.

### Wiki-style internal linking

Vim's built-in `gf` opens the file under the cursor. Two settings extend this for a Markdown wiki:

```vim
set suffixesadd+=.md
setlocal suffixesadd+=.md
set path+=/home/dhcolmenares/Documents/nvdata
noremap gS :e <cfile><.%:e<CR>
```

`suffixesadd+=.md` lets `gf` resolve bare names (e.g. `[[week]]`) by trying `week.md`. `path+=…/nvdata` extends the search path to include the wiki directory. `gS` is a custom variant: it opens the file under the cursor with the *current file's extension* appended — useful when working in a `.md` wiki where I want every newly-created link to inherit `.md` automatically. As an exaptive side effect, it also creates a file named after a BibTeX key, which is how I attach annotation files to bibliographic entries.

### FZF search and content grep

```vim
nnoremap <C-p> :Files ~/Documents/nvdata/<Cr>
nnoremap <C-k> :Ack -i <C-R><C-W><CR> ~/Documents/nvdata/

let g:fzf_preview_window = 'right:50%'

```

`Ctrl-P` runs `:Files` over the wiki directory: fuzzy-find any file by name. `Ctrl-K` runs `:Ack` (using The Silver Searcher, `ag`) for the word under the cursor, scoped to the wiki. `:Tags` and `:FindTag` use ripgrep to find inline tags marked with the convention `[#### tag-name ]`.

### Concealment for nested lists

A small visual tweak that materially changes how outlines look:

```vim
augroup NestedListConceal
    autocmd!
    autocmd FileType markdown,pandoc call SetupNestedListConceal()
    autocmd BufEnter *.md,*.markdown call SetupNestedListConceal()
augroup END

function! SetupNestedListConceal()
    silent! call clearmatches()
    call matchadd('Conceal', '^\s\{8,\}\zs[-*+]\ze\s', 30, -1, {'conceal': '◆'})
    call matchadd('Conceal', '^\s\{4,7\}\zs[-*+]\ze\s', 20, -1, {'conceal': '○'})
    call matchadd('Conceal', '^\s\{0,3\}\zs[-*+]\ze\s', 10, -1, {'conceal': '◉'})
endfunction

set briopt+=list:2
```

The Markdown source is plain `- item`, perfectly portable to any other tool. On screen, top-level bullets render as `◉`, second-level `○`, third-level `◆`. Combined with `briopt+=list:2` for hanging indentation of wrapped lines, outlines start to look like outlines. The three priority levels in `matchadd` ensure that the most-indented match wins.

### Placeholder hunting

```vim
match Error '{\w\+}'
nnoremap ,ph :vimgrep /{\w\+}/ %<CR>:copen<CR>
```

While drafting, I leave `{TODO-FIND-CITATION}` and `{CHECK-DATE}` markers wherever something needs to be filled in later. The first line highlights any `{word}` token as an error; the second collects every such token in the current file into the quickfix window. The pre-submission sweep becomes mechanical: open quickfix, walk the list, resolve each one.

## 5. Ranger

![Ranger in a floating Alacritty window with a JPEG previewed as colored ASCII](images/ranger.png)
*Ranger floating, with a `jp2a` color preview in the right pane.*

For directory-level browsing — operations beyond what `rf` covers, such as moving into a directory, batch-renaming, copying between locations — I use Ranger. The Sway keybinding spawns it in a floating Alacritty:

```sway
bindsym $mod+alt+o exec $term_float -e ranger
```

Image previews under Wayland are notoriously fragile. The traditional X11-only options (`w3mimgpreview`, `ueberzug`) do not work; the maintained Wayland fork `ueberzugpp` works but adds dependencies. The simpler approach is the same one `rf` uses: render images as ANSI-colored text via `jp2a`. In `~/.config/ranger/scope.sh`:

```bash
image/*)
    jp2a --width="$PV_WIDTH" --colors "$FILE_PATH" && exit 4
    exit 1;;
```

Exit code `4` instructs Ranger to display the script's stdout in the preview pane verbatim, including any ANSI escape sequences. In `~/.config/ranger/rc.conf` two settings need to be set explicitly:

```
set preview_images false
set preview_colorized true
```

The first disables Ranger's own image-rendering machinery (which would otherwise attempt to use a non-functional protocol). The second permits ANSI escape codes through the preview pane. The combination produces colored ASCII previews of every image in the current directory.

Ranger's keybindings are essentially Vim's by default, so no further customization is required.

## 6. Newsboat

![Newsboat showing a feed list with cyan-yellow-blue color scheme and unread counts](images/newsboat.png)
*Newsboat. Vim keybindings, terminal-native rendering, ~100 feeds.*

Newsboat handles RSS for somewhere over a hundred feeds — academic blogs, journals, news sites, personal blogs. The full configuration:

```
# Interactions
browser "rdrview -B lynx"
external-url-viewer "urlscan -dc -r 'firefox {}'"
auto-reload yes
feed-sort-order unreadarticlecount-asc
text-width 120
articlelist-format "%4i %f %D  %?T?|%-17T| ?%t"
goto-next-feed no

# Colors
color info default default reverse
color listnormal_unread cyan default
color listfocus blue default reverse bold
color listfocus_unread blue default reverse bold
color article cyan default
```

The single most consequential line is the browser definition:

```
browser "rdrview -B lynx"
```

`rdrview` is a Readability-style content extractor — a CLI port of the same algorithm that powers Firefox Reader View. It identifies the main content of a webpage and strips everything else (sidebars, navigation, cookie banners, autoplay video, advertising chrome). The `-B lynx` flag pipes the extracted content into Lynx, the long-standing text browser. Pressing `o` on any RSS item produces a clean text rendering of the actual article in the same terminal. Most blog posts and news articles read better this way than they do on the actual website. A few read better than they did when their authors published them.

Two macros handle exceptions. `,v` opens a video URL in `mpv` directly (useful for video feeds and lecture series); `,r` opens an article in `rdrview`'s own pager rather than Lynx, which is sometimes preferable for longer pieces. The `external-url-viewer` line uses `urlscan` to enumerate every URL in the current article and pipe a chosen one to Firefox — an escape hatch for image-heavy posts and interactive longreads.

The Vim keybindings replicate `h`/`j`/`k`/`l` navigation across all three panes (feedlist, articlelist, article). `J`/`K` jump between feeds. `G`/`g` and `d`/`u` follow Vim conventions for end/home and half-page paging.

Newsboat is keybound to `Mod+Alt+R` in Sway:

```sway

```

## 7. Email: aerc, matcha, cmdg

I run three terminal email clients, each handling a different account or use case.

**`cmdg`** is a Gmail-specific TUI written in Go, used for my UCI institutional account. Because Gmail enforces OAuth, `cmdg` requires its own configuration file with credentials and refresh tokens — that this works at all is a small miracle of someone's persistence against an authentication flow that does not want it to. It also depends on `EDITOR` and `PAGER` environment variables to launch composition and reading windows, which is why every `cmdg` invocation in my Sway config goes through `bash -l -c`:

```sway
bindsym $mod+alt+m exec alacritty -e bash -l -c "/usr/local/bin/cmdg -config ~/.cmdg/cmdg-uci.conf"
bindsym $mod+alt+p exec foot -e bash -l -c "/usr/local/bin/cmdg"
```

A login shell guarantees that `~/.profile` has been sourced and that `EDITOR=vim` and `PAGER=less` are set before `cmdg` starts. Without it, `cmdg` falls back to defaults that may or may not exist on the system, and composition silently fails. The two bindings differ by config path: `Mod+Alt+M` opens the UCI account using `cmdg-uci.conf`; `Mod+Alt+P` opens the personal Gmail account using `cmdg`'s default config path. The personal binding deliberately uses `foot` rather than Alacritty to mark the visual distinction between the two accounts at a glance.

**`matcha`** is the TUI I use for non-Gmail personal mail. It is built on Bubble Tea — Go's lovely modern TUI framework — and represents the kind of terminal mail client that should exist by default in 2026: clean three-pane layout, sensible keybindings, almost no configuration required to get a working inbox. It is the antidote to Mutt, the venerable but labyrinthine alternative whose `muttrc` is the kind of file you do not so much *write* as *inherit* from a sysadmin uncle. `matcha` is invoked through the Lurking deck:

```
1 | ELECTRONIC MAIL    | Fetch personal messages    | alacritty -e bash -l -c "matcha"
```

The same `bash -l -c` wrapper applies for the same reason as `cmdg`: a fully-loaded environment for any subprocess that depends on `EDITOR`, `PAGER`, IMAP credentials in the keyring, or shell aliases.

**`aerc`** is the third client, used selectively when I need its more featureful interface — particularly its threaded view, its embedded HTML rendering through `w3m`, its support for multiple accounts in tabbed buffers, and its scriptable filters for processing incoming mail. `aerc` reads its configuration from `~/.config/aerc/`, where `accounts.conf` holds IMAP/SMTP details per account, `aerc.conf` holds global UI settings, and `binds.conf` holds keybindings (which I keep as close to Vim as the client allows). Like `cmdg` and `matcha`, it benefits from a login-shell launch when invoked from a window-manager keybinding.

All three clients share a property worth naming: composition happens in Vim, opened in-place by the client through `$EDITOR`. Whatever email I send, in whichever account, is written in the same editor with the same configuration as everything else in this post. The settings, mappings, spell-checkers, and dictionaries follow me into the message buffer.

## 9. How the parts compose

The bindings on Sway organize the pieces into a small, memorable namespace:

| Keybinding      | Launches                                    |
|-----------------|---------------------------------------------|
| `Mod+Alt+L`     | Lurking deck (in a fresh Alacritty)         |
| `Mod+Alt+V`     | Vim                                         |
| `Mod+Alt+O`     | Ranger (floating)                           |
| `Mod+Alt+R`     | Newsboat (floating)                         |
| `Mod+Alt+M`     | cmdg, UCI Gmail account                     |
| `Mod+Alt+P`     | cmdg, personal Gmail account                |
| `Mod+Alt+N`     | nv.sh (notational velocity scratch)         |
| `Mod+Alt+F`     | Firefox                                     |

Inside any terminal, the Lurking deck adds a second namespace: digit keys for items 1 through 6, letter keys for the few that have them. Inside Vim, the `,` and `'` leader keys add a third. Together they cover every common operation with at most three keystrokes and no mouse.

The pattern that recurs at every level is the same: a TUI tool with Vim-style keybindings, configured to do one thing well, launched in a floating Alacritty window through a login shell. The entire environment fits in a handful of plain-text configuration files — `~/.vim/.vimrc`, `~/.config/lurking/items`, `~/.config/ranger/rc.conf` and `scope.sh`, `~/.config/newsboat/config`, `~/.config/zathura/zathurarc`, `~/.config/sway/config`, the various email client configs, and the Dávila aphorism file. All of them are versioned in a `dotfiles` repository. A new machine becomes a working machine in a single afternoon.

## 10. A short manifesto, since you asked

There is a temptation, when describing a setup like this one, to disclaim it. To say something like *of course this is not for everyone* or *of course modern tools have their place*. I would rather not. The terminal — text-based, keyboard-driven, Ncurses-rendered, composable through pipes — is, on every axis I care about, superior to every paradigm that came after it. The argument has several parts.

**Aesthetics is tantamount.** The visual idiom of a working environment shapes the kind of attention you bring to it. I write differently in front of monospace amber on black than I do in front of a Word document, and the difference is not trivial. A 1980s phosphor terminal asks you to slow down, to think before typing, to commit to a sentence before you can backspace it. That is exactly the right disposition for prose. The slow, deliberate pleasures of a CRT-flavored research environment are partly nostalgic — I was born in 1979 and grew up among machines like these — but the underlying interaction model was already mostly right by 1985. The problem with everything since has been a gradual accretion of distractions.

**Ncurses should be mandatory.** Anything that draws to a terminal cell grid in a way the user can predict and manipulate is a candidate for inclusion in this setup; anything that requires GPU compositing, Electron, or a 200-megabyte runtime is, by definition, not. (I will admit one exception: Bubble Tea, the Go TUI framework that powers Matcha and a growing number of newer clients, is wonderful too. Anything that lets developers ship Ncurses-quality interfaces without writing C is fine by me.)

**The mouse, and the use of images, are banned by Deuteronomy.** This is partly a joke and partly not. There is an iconoclastic streak running through the whole setup — a refusal of images in favor of text, of GUIs in favor of keystrokes, of buttons in favor of letters — and the rendering of JPEGs as colored ASCII through `jp2a` is its sincerest expression. The mouse is the graven image of human-computer interaction. It produces nothing the keyboard cannot, slower and with more wrist strain. Vim's whole thesis, philosophically, is that the keyboard is sufficient; after enough years inside it, you stop noticing the mouse exists.

**Email providers should be more welcoming to command-line clients.** Google in particular. The fact that `cmdg` works, that `matcha` works, that `aerc` works at all in 2026 is despite the platforms, not because of them. The OAuth gauntlet, the periodic token expirations, the app-password kludges — none of this needs to exist. Mutt remains the canonical TUI mail client, but its configuration is a labyrinth in the technical sense: you can get lost in it. Matcha is the antidote — useful on the second day rather than the second month — and the existence of more clients in its mold is something I would happily campaign for.

**The death of Infocom in 1989, and of the parser-driven text adventure more broadly, was a world tragedy comparable in magnitude only to the dissolution of the Soviet Union later that same year.** Both events were the foreclosure of certain imaginative possibilities. *The Lurking Horror* is one of the strangest, finest objects ever produced by a commercial software company: an MIT-set Lovecraftian text adventure that takes itself entirely seriously, written in a parser-driven prose more careful than most novels of its decade. That an entire genre died because it could not compete with the *visual* thrill of *Doom* is a fact I have not entirely forgiven the 1990s for, and the setup described above is, among other things, a small tribute to that lost world. Lovecraft would have approved of an interface in which the unspeakable is, by mechanical necessity, written.

## 11. Roadmap

A few pieces I have not yet pulled into the terminal but intend to.

**News, exclusively through RSS.** Newsboat already covers the bulk of my reading, but a few news sites remain stubbornly outside the feed. The aim is to absorb everything: every newspaper, every academic blog, every aggregator. Where official feeds do not exist, RSS-Bridge and `rss-please`-style scrapers can synthesize them. The endpoint is to never visit a news website manually again. (The web is, when you stop visiting it, surprisingly easy to live without.)

**Music in the terminal.** `ncspot` is a Rust-based TUI client for Spotify, using the official Connect API. The interface is essentially Vim-like; playback is native; the resource footprint is a fraction of the official Spotify desktop client, which, like every other Electron app, is several hundred megabytes of Chromium pretending to be a music player. I see no reason not to switch.

**WhatsApp in the terminal.** This one is harder, because Meta does not officially support third-party clients. The pragmatic options are `nchat` — a Linux TUI that supports several messaging protocols including WhatsApp through reverse-engineered bridges — and `matterbridge`, which can pipe WhatsApp through IRC or XMPP, both of which have excellent terminal clients. Either approach is more fragile than I would like, but the alternative — keeping a Chrome tab open all day for WhatsApp Web — falls foul of the Deuteronomy clause above.

The general principle: every application that lives outside the terminal is a candidate for being moved into it. The destination is a single Sway workspace, no browser permanently open, no graphical client whose absence would be felt. We are not there yet. We are getting closer.

— D.H.C.
